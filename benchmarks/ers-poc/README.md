# ERS-PoC

A benchmark to evaluate the end-to-end efficacy of ERS semantic compression.

Automatically generated report in [results/report.md](./results/report.md)

## Concept

Natural language incurs high token costs. ERS-PoC evaluates the viability of replacing source text with ERS by measuring the trade-off between token reduction and accuracy loss during QA inference.

## Architecture

* **Dataset**: [alxraun/ers-bench-nano](https://huggingface.co/datasets/alxraun/ers-bench-nano) - parallel QA over `source` vs `ers_artifact`.
* **Candidate**: Extracts facts solely from context and reconstructs the answer in natural language.
* **Judge**: `GEval` — compares the candidate's answer against the Ground Truth using logical alignment criteria, ignoring differences in abstraction level, style, and the presence of extra details.

## Metrics

* **CR** (Compression Ratio): `(1 - (ers_tokens / src_tokens)) * 100` - context reduction.
* **PR** (Performance Retention): `(avg_ers_score / avg_src_score) * 100` - retained answer correctness.
* **Δ** (Absolute Delta): `ers_score - src_score`.

## Quick Start

It is recommended that [uv](https://github.com/astral-sh/uv) be installed.

1. **Environment**:
   ```bash
   cp .env.example .env
   ```

2. **Execute** (Generate, Evaluate, and Report):
   ```bash
   ./run.sh run-all --models "openrouter/openai/gpt-oss-20b" --judge-model "openai/gpt-5.4"
   ```

## Commands

* `generate`: Fetch responses from candidate models.
* `evaluate`: Grade responses against Ground Truth.
* `report`: Aggregate metrics and output Markdown reports.
* `run-all`: Sequential execution of the full pipeline.

## Known Limitations & Empirical Observations

### 1. Methodological Facts
* ERS artifacts and QA pairs were generated independently, using only `source.txt` as input, without access to each other's output.
* ERS artifacts and QA pairs, including `complex` questions, were generated by the same frontier model.

### 2. Empirical Observations
* ERS artifacts in the dataset are the result of lossy generation: structural and causal relationships are preserved, while narrative elements and some low-level identifiers are dropped.
* For some `simple` questions that require direct fact retrieval, the ERS artifact may drop certain narrative elements and some low-level identifiers.
* For some `complex` questions that require multi-hop reasoning over the original `source`, the ERS artifact may already contain the explicit logical chain (e.g., `A -> B => C`), turning the task into direct fact retrieval.

### 3. Possible Interpretations
* ERS generation may systematically preserve causal chains while abstracting low-level identifiers into higher-level concepts. This potentially explains why `complex` questions suffer less from compression, while `simple` questions sometimes lose accuracy: the former more often rely on relationships, the latter more often rely on specific names.
* It is plausible that small models (4B) successfully extract answers when the logical link is already explicit in ERS, but may struggle when independent sorting or linking of disconnected nodes is required.
* Since ERS and QA were generated by the same model, its internal attention mechanisms may have highlighted the same structural nodes. This may have led some ERS artifacts to contain direct answers to questions that were designed to be `complex`.
